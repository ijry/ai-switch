# GitHub Actions Cross-Platform Release Design

## Goal

Add a manual GitHub Actions release workflow that builds and publishes cross-platform AI Switch binaries from `main`.

The workflow creates one GitHub Release from user-provided inputs and uploads desktop installers, updater assets, the standalone server binary, and the Tailscale sidecar binary for Windows, macOS, and Linux.

## Trigger

The release workflow uses `workflow_dispatch` only.

Inputs:

- `tag`: required release tag, for example `v0.1.0`.
- `release_name`: optional display name. Defaults to `tag` when empty.
- `draft`: boolean, default `true`.
- `prerelease`: boolean, default `false`.

Manual triggering keeps release publication deliberate while still automating all build and upload work.

## Release Jobs

The workflow uses a platform matrix:

- `windows-latest`
- `macos-latest`
- `ubuntu-latest`

Each matrix job installs Node, pnpm, Rust, Go, and platform-specific Tauri dependencies. It then runs validation before packaging:

- `pnpm typecheck`
- `pnpm test:run`
- `pnpm rust:check`
- `pnpm rust:test`
- `go test ./...` in `sidecar/ai-switch-tsnet`

After validation, each job builds:

- Tauri desktop bundle for the current platform.
- `ai-switch-server` release binary for the current platform.
- `ai-switch-tsnet` sidecar binary for the current platform.

The sidecar binary is written to the Tauri `externalBin` path expected by `src-tauri/tauri.conf.json` before `pnpm tauri:build` runs.

## Desktop Bundle Targets

`src-tauri/tauri.conf.json` currently targets only Windows `nsis`. The release implementation should allow cross-platform bundles:

- Windows: `nsis`
- macOS: `dmg`
- Linux: `deb` and `appimage`

The existing updater endpoint remains:

```text
https://github.com/ijry/ai-switch/releases/latest/download/latest.json
```

## Signing

The workflow requires Tauri updater signing secrets:

- `TAURI_SIGNING_PRIVATE_KEY`
- `TAURI_SIGNING_PRIVATE_KEY_PASSWORD` optional, depending on the private key

If the private key is absent, the workflow must fail before publishing release assets. Publishing unsigned updater assets would make the in-app updater unreliable.

## Release Creation And Upload

One release is created for the requested tag. Platform jobs upload their artifacts to that release.

Asset names include the application version, OS, and architecture to avoid collisions. The workflow should upload:

- Platform desktop installers and updater artifacts generated by Tauri.
- `latest.json` updater manifest when generated by Tauri.
- Standalone `ai-switch-server` archive per platform.
- Standalone `ai-switch-tsnet` archive per platform.

The first implementation can keep `draft` defaulting to `true` so failed manual review does not publish broken assets.

## Failure Behavior

The workflow fails fast for missing signing secrets, failed tests, or missing expected artifacts. It should not create a public release unless all required jobs complete successfully.

If a tag already exists, the workflow should reuse it for release creation rather than trying to force-update git history.

## Out Of Scope

- Store-specific signing, notarization, or app-store distribution.
- Automatic releases from `main` or tags.
- Changing the updater URL or adding release channels.
- Removing the existing manual local build commands.

## Acceptance Criteria

- A maintainer can run the release workflow manually from GitHub Actions.
- The workflow builds Windows, macOS, and Linux desktop packages.
- The workflow uploads release assets for desktop, standalone server, and sidecar binaries.
- The workflow fails if updater signing is not configured.
- Existing local development commands keep working.
